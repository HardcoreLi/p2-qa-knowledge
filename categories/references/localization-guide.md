# 本地化逻辑与 Bug 分配

## 简答
> 本地化 Bug 分配：显示本地化 Key → 给策划；静态文本问题 → 给 GUI；动态文本显示错误 → 给客户端。不好区分且没显示 Key 的都给客户端。

---

## 本地化工作流程

```
策划/运营配置 Key
       ↓
客户端/GUI 使用 Key
       ↓
如缺少翻译 → 显示 Key
       ↓
策划上传本地化配置即可（不需要客户端/GUI 返工）
```

---

## 文本类型分类

### 1. 静态文本
- **定义**：完全不变的文本，如某些标题
- **负责人**：**GUI**

### 2. 动态文本
- **定义**：根据数据/状态变化的文本
- **负责人**：根据问题类型分配

---

## Bug 分配规则

| 问题表现 | 分配给 | 说明 |
|----------|--------|------|
| 显示了**本地化 Key** | **策划** | 策划上传本地化配置即可 |
| **静态文本**有问题 | **GUI** | 如标题等完全不变的文本 |
| **动态文本**显示错误内容 | **客户端** | 显示了错误的文本 |
| 不好区分，但**没有显示 Key** | **客户端** | 兜底规则 |

### 快速判断流程

```
是否显示了本地化 Key？
    ├── 是 → 给策划
    └── 否 → 是静态文本吗？
              ├── 是 → 给 GUI
              └── 否/不确定 → 给客户端
```

## 常见坑点

- **显示 Key 就给策划**：这是最明确的判断标准
- **静态 vs 动态**：不确定时，没显示 Key 就给客户端
- **不需要返工**：策划上传配置后，客户端/GUI 不需要重新修改
